## 1. Device Characteristics

| ID | Type | Characteristic |
|---|---|---|
| SAI-EF-CHAR-001 | Safety | The device is used in outpatient dermatology clinics, primary care practices, and home environments. |
| SAI-EF-CHAR-002 | Safety | The device is used by adult patients (>=18 years) with a confirmed Cox-Dewar Dermatitis (CDD) diagnosis. |
| SAI-EF-CHAR-003 | Safety | The device supports patient-initiated skin image capture from home using smartphone cameras. |
| SAI-EF-CHAR-004 | Safety | The device provides warnings to users when image quality is inadequate for reliable IDM calculation. |
| SAI-EF-CHAR-005 | Safety | The device provides clinical alerts through multiple configurable channels (in-app, email, SMS) to healthcare professionals when IDM values exceed predefined thresholds. |
| SAI-EF-CHAR-006 | Safety | The device allows healthcare professionals to set patient-specific alert thresholds. |
| SAI-EF-CHAR-007 | Safety | The device is used by qualified healthcare professionals (dermatologists, primary care physicians) to support monitoring of disease activity. |
| SAI-EF-CHAR-008 | Safety | The device requires healthcare professional interpretation of IDM values and is not intended for standalone diagnostic use. |
| SAI-EF-CHAR-009 | Safety | The device clinical alerts, within 1 minute of threshold exceedance, through multiple configurable channels to healthcare professionals. |
| SAI-EF-CHAR-010 | Security | The device operates on iOS and Android mobile operating systems for patient access. |
| SAI-EF-CHAR-011 | Security | The device operates with web browsers for clinical interface access. |
| SAI-EF-CHAR-012 | Security | The device operates with cloud backend infrastructure for data processing and storage. |
| SAI-EF-CHAR-013 | Security | The device handles and stores confidential patient health data, including skin images, IDM values, and associated clinical information. |
| SAI-EF-CHAR-014 | Security | The device sends and receives data on IT networks (mobile networks, internet) between mobile applications, clinical interfaces, and cloud backend services. |
| SAI-EF-CHAR-015 | Security | The device integrates with external services for alert delivery (e.g. SMS, email). |

## 4. Hazardous Sequences - Risk Mitigation

### A Note on Risk Mitigation Methods

In accordance with ISO 14971, risk controls have been defined in the following methods in this order of priority:

1. Inherently safe design
2. Protective measures
3. Information for safety

For some risks, inherently safe design is not feasible. The rationale for this is as follows: the device's intended function relies on an ML-based prediction algorithm, and prediction outputs are inherently probabilistic rather than fully deterministic. As a result, it is not technically possible to eliminate the possibility of inaccurate predictions through design alone. In these scenarios, protective measures and/or information for safety are implemented.
