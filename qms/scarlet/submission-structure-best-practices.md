# Technical Documentation: Best Practices for Submission Structure

The EU Medical Device Regulation (Regulation (EU) 2017/745) sets clear expectations for technical documentation to be structured to facilitate assessment.

EU MDR, Annex II:

> "The technical documentation and, if applicable, the summary thereof to be drawn up by the manufacturer shall be presented in a clear, organised, readily searchable and unambiguous manner"

## Core Principles

The following are the core principles of crafting and structuring technical documentation within an EU MDR certification application:

- **Organisation:** Present information in a logical order and present it using naming conventions and titles that ensure it can be easily mapped to specific regulatory requirements.
- **Searchability:** Provide navigation aids such as tables of contents, data identifiers, traceability matrices, and hyperlinks. Use searchable file formats to enable assessors to quickly locate evidence of compliance.
- **Clarity:** Use plain, professional language, avoiding jargon or undefined acronyms. A trained assessor should easily understand documentation without requiring in-depth product knowledge.
- **Unambiguity:** Maintain consistency in terminology, device identifiers, processes, and conclusions across all documents. This eliminates confusion and prevents delays during assessment.

These principles matter for the following reasons:

- **For Manufacturers:** Supports comprehensive documentation and enables effective gap analysis. Provides a clear demonstration of compliance, improves efficiency and simplifies future updates.
- **For Conformity Assessment Bodies:** Facilitates efficient assessment, reduces the need for follow-up questions, and accelerates approvals.

## Best Practices

The following are best practices for creating high-quality, structured technical documentation for your application submission.

### Documentation Organisation

The overall organisation of your technical documentation is crucial for clearly demonstrating compliance with the applicable regulatory requirements. Under EU MDR, Annex I sets out the General Safety and Performance Requirements (GSPR). Manufacturers must demonstrate conformity with every applicable requirement. This can only be achieved through well-structured and traceable technical documentation.

Well-organised technical documentation must contain the following:

- Uniquely referenced files or sections that clearly align with each applicable GSPR or ER.
- A GSPR or ER checklist that defines the regulatory requirements, specifies applicability to the device, and maps each requirement to the supporting evidence in the technical file.

Such checklists are not optional; they are essential for enabling manufacturers to perform practical internal gap analysis and providing assessors with a clear roadmap through the documentation. To understand more, see the GSPR checklist article.

There is no single gold standard for formatting technical documentation. However, this recent IMDRF proposal for structuring medical device submissions offers a state-of-the-art benchmark that manufacturers can use to evaluate and refine their own document organisation.

### Searchability - Expectations

It is essential that an assessor can easily navigate and search through your documentation to assess your application efficiently.

Appropriate digital file formats should be used that support the following searchability:

**Comprehensive traceability** -- Your technical documentation must demonstrate clear, end-to-end traceability from the intended purpose through requirements, design, risk management, verification, and validation, all the way to evidence of clinical safety and performance. This can be achieved by:

- Providing standalone traceability matrices that use unique identifiers to link items across the technical documentation.
- Embedding cross-references within documents so the assessor can move seamlessly between related sections.

See the need for clear traceability article for a more detailed explanation of traceability expectations.

**Structured navigation** -- Documentation should be logically organised for easy navigation. This includes:

- Consistent and descriptive naming of directories, files, and section headers.
- Inclusion of tables of contents, indexes, and other navigation aids that provide a roadmap for reviewers.

**Searchable formats** -- Files must support text-based searching to allow rapid access to key terms, data, and references. To ensure this:

- Use machine-readable file formats (e.g., PDF, Word, Excel). Avoid scanned PDFs that do not support keyword search.
- Incorporate diagrams and schematics (e.g software architecture) to clarify complex information. However, avoid relying heavily on images as they can limit searchability.

**Cross-referencing and linking** -- Regulatory submissions should integrate cross-references and hyperlinks to both internal documents and, where appropriate, external resources. This allows assessors to verify evidence and supporting material without unnecessary searching.

### Searchability - Modern Tooling

If using modern documentation management systems and software development tooling, ensure that cross-references and identifiers remain intact when exporting to regulatory submission formats.

Many modern documentation management systems provide cross-referencing and traceability between discrete technical documentation items. This is advantageous to the manufacturer when constructing the technical documentation to ease searchability, confirm comprehensiveness and perform gap analysis before submission. Unfortunately, when exporting from these systems to standard document formats such as PDF, Word or Excel, it is common for this cross-referencing and traceability to be lost. As such, the assessor does not benefit from the searchability inherent in the native format during assessment.

Similarly, many development teams produce machine-generated software development documentation, such as code analysis reports, test execution reports, and software bill of materials. Converting these formats to standard documentation formats can be cumbersome and suboptimal for assessment.

### Clarity and Consistency

- Define acronyms and technical terms when first used and provide a glossary.
- Align terminology across the components of the technical documentation (e.g. GSPR/ER checklist, labelling, IFU, and risk documentation). Inconsistencies in terminology are a common reason for confusion and delay.
- Structure the documentation so that it is clear how regulatory activities were planned and executed and what the outcomes were. Ensure consistency across these areas and provide clear traceability of the development journey.

## Conclusion

Clear, organised, readily searchable, and unambiguous technical documentation is not just a regulatory expectation; it is a cornerstone of efficient compliance. By structuring documentation around the GSPR (EU MDR) or Essential Requirements (UK MDR) and supporting these with comprehensive checklists, as well as using appropriately searchable file formats, manufacturers create a transparent and traceable compliance story.

For manufacturers, this means fewer gaps, more efficient updates, and a stronger foundation for lifecycle management. For conformity assessment bodies, it enables timely, effective reviews and accelerates market access.
