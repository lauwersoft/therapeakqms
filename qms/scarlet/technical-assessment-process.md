# Scarlet's Technical Assessment Process

The technical assessment process evaluates your technical documentation against applicable regulatory requirements:

- **Technical documentation review:** Scarlet will conduct an assessment of your technical documentation. This review is a requirement for customers wishing to place a CE or UKCA mark on their product (if their product is Class Im, Class IIa, Class IIb, or Class III).
- **Query process:** During the assessment, Scarlet may raise queries -- requests for more information that are objectively based on regulatory requirements. All queries are provided via the Scarlet web app, and you are required to provide a query acknowledgement for each query raised.
- **Assessment outcomes:** Following a technical documentation review, Scarlet may either issue a technical documentation assessment certificate (subject to final review and decision making process) or refuse to grant certification and provide an explanation for that decision.

The assessment process may vary depending on whether this is your initial certification or you are adding new devices, and on the risk classification of your devices.

## Best Practices for Responding to Queries

When responding to queries raised during your technical assessment, consider the following:

1. **Understand the query:** Provide an opening statement that defines your understanding of the query raised and outlines your approach to resolving it. Read all references to medical device standards clauses that Scarlet has provided.
2. **Provide evidence:** Include new or existing evidence in your query acknowledgement, accompanied by clear rationale for why this evidence is sufficient to satisfy the query. Remember that assessors can only review documentation that has been submitted.
3. **Consider downstream impacts:** Review the downstream consequences of any changes you make. The information you provide should make sense in the wider context of the rest of your documentation.
4. **Make evidence easy to find:** Provide clear links to evidence. It should be easy for assessors to search and locate the information requested in queries.
5. **Address comprehensively:** Ensure you address all parts of the query. Where examples are given, note that language like "includes but is not limited to" indicates you should fix the query in the example as well as the wider context across your submission.

More information about how to respond to queries can be found in the Deep Dive section below.

## Common Pitfalls to Avoid

- Misinterpreting the query -- if unclear, seek clarification before investing effort in resolution
- Making claims without supporting evidence being uploaded
- Making evidence difficult to find or not correctly linking across documents
- Inconsistent claims across documentation
- Making changes without considering downstream impacts
- Only partially addressing the query
- Not describing all instances where changes have been made

## Sampling Considerations for Multiple Devices

If you are seeking certification for multiple medical devices:

- **Initial certification:** Each Class III device requires a separate technical documentation assessment certificate. For other classes, a sampling plan will be drafted, and at least one representative device per device group (Class IIb) or category (Class IIa) will be assessed before the QMS certificate is issued.
- **Adding new devices:** If you have an existing QMS certificate, you must inform Scarlet of changes. Devices within the current device range are added to the sampling plan -- no new application is required for Class IIb and Class IIa devices.
- **Sampling plans:** Scarlet is required to have a sampling plan for each QMS certificate covering more than one device. For EU MDR, at least 15% of devices under each risk category and EMDN code must be assessed over the certificate validity period. Scarlet prioritises devices with novel technology (especially AI), higher inherent risk, and those not yet assessed.

## Deep Dive: Best Practice for Responding to Queries

### What Is a Query?

- Queries are raised as part of your technical submission assessment.
- Scarlet will provide you with all queries via the Scarlet web app.
- A query is a request for more information. It is not innately pushing towards any specific action.
- Queries are objectively based on regulatory requirements. In most cases, Scarlet will include references to medical device standards clauses that relate to the query raised.
- You are required to provide a query acknowledgement for each query raised.

### Query Acknowledgements

- Query acknowledgements should address any questions or concerns raised in the query.
- Scarlet will request responses to all queries raised through the Scarlet web app.
- You are welcome to provide any additional information, justification or supporting evidence as part of your query acknowledgement.

### What Are Scarlet Reviewing When Considering Your Query Acknowledgements?

- Have you correctly understood the query raised and the associated regulatory clauses?
- Have you stated how you intend to address the query?
- Have you pointed to new, or existing, evidence that addresses the query?
- Have you provided rationale for how this evidence adequately addresses the query?
- Has the query been comprehensively addressed across your technical documentation?
- Does the query acknowledgement raise any additional queries?
- How well your response demonstrates regulatory compliance
- Completeness and cohesion across your submission

### General Advice and Considerations When Responding to Queries

- Provide an opening statement that defines your understanding of the query raised and outlines your approach to resolving it.
- Read all references to medical device standards clauses that Scarlet has provided and ensure you consider the content of these clauses when determining how best to address the query.
- Include new or existing evidence in your query acknowledgement. Accompany this evidence with statements that provide a clear rationale for why you believe this evidence is sufficient to satisfy the query.
- Remember that our assessors can only review the documentation that has been submitted. It is possible that you have the information we are requesting, but if it was not provided in the submission then assessors will not know that it exists. You may therefore need to provide that documentation in a query acknowledgment.
- When considering how to address a query, be sure to review the downstream consequences of any changes that you make or information that you provide. The information that you provide should still make sense in the wider context of the rest of your documentation.
- Make sure to provide clear links to evidence. It should be easy for assessors to search and look for the information requested in queries.
- Where an example is given in a query, please note the language used -- "includes but is not limited to". This indicates that you should ensure that the query is fixed in the example as well as the wider context across your submission.
- **Clinical example:** Scarlet may raise a query regarding the clinical benefits. In this scenario, consider how your response to the query might have cascading effects, potentially impacting the literature search, clinical evaluation activities, risk analysis, and other related areas.
- **Software example:** Scarlet may raise a query regarding a specific software requirement. In this scenario, consider how your response to the query might have cascading effects, potentially contradicting another software requirement, or rendering the corresponding testing activity no longer adequate to the updated requirement.

### Common Pitfalls with Query Acknowledgements

- **Misinterpreting the query.** Whilst Scarlet's queries aim to be clear, concise and objective, all communication is open to misinterpretation. Thoroughly read the query and any medical device regulation clauses provided. If the query is still unclear, seek clarification from Scarlet before investing effort in the resolution.
- **Making claims with a lack of supporting evidence being uploaded.**
- **Making evidence difficult to find.** For example not correctly linking across documents or providing evidence that is difficult for assessors to find.
- **Inconsistent claims across the documentation.** The documentation should be consistent throughout.
- **Making changes without considering the downstream impacts.** If you make changes in response to a query, be sure to consider the implications across your technical submission.
- **Only partially addressing the query.** Make sure to address all parts of the query and be mindful that examples given may not cover ALL instances of a query that need addressing.
- **Not describing all instances where changes have been made.** This can make cross-referencing difficult for our assessors.

### Example Query and Acknowledgement

The following is an example query that Scarlet may send regarding a technical assessment and an example of an appropriate acknowledgement for that query.

#### Example Query

**Query Name:** Lack of software update requirements

**Created on:** 15-01-2025

**Statement:** Scarlet requests clarification, as it is unclear that software update requirements have been defined.

**Supporting Details:**

Scarlet observes that software update requirements have not been defined. As a result, it is unclear whether requirements to support:

- upgrades from previous versions, including maintaining data integrity, and compatibility with prior versions
- rollback to the previous version after upgrade;

[IEC 82304-1:2016; 4.2.f]

**Evidence files:**

- Use Requirements Document.pdf

#### Example Query Acknowledgement

Customer acknowledges that software update requirements are not adequately defined within the provided Use Requirements Document. The following updates address this issue:

- The Use Requirements Document has been updated to include new use requirements.
- The Software Requirements Document has been updated to include new software requirements.
- Additional verification activities have been conducted to verify the new software requirements. These activities are outlined in the Verification Plan and Test Specifications files. The verification execution and outcomes are documented in the Test Execution Reports and Verification Summary Report files.
- Additional validation activities have been conducted to validate the new use requirements. These activities and executions are outlined in the Validation Plans and Validation Reports files.

**Files adjusted:**

- Use Requirements Document.pdf
- Software Requirements Document.pdf
- Verification Plans.pdf
- Test Specifications.pdf
- Test Execution Reports.pdf
- Verification Summary Report.pdf
- Validation Plans.pdf
- Validation Reports.pdf

For details on the updates to all files, please consult the documents' change history section.
